## TL;DR
A subregion is a logical region that's a child of another — produced by partitioning. Its `index-space.md` is the index-subspace at one color of the parent's `IndexPartition`; it shares the parent's `field-space.md`. Subregions are themselves full-fledged `logical-region.md`s — you can take privileges on them, partition them further, and pass them in region requirements. The confusion: a subregion is not a slice or a view — it's a first-class region whose data the runtime materializes (and shares with the parent) on demand. Subregion-of-subregion hierarchies are normal and inexpensive.

## Mental model
Subregions are SQL views generated by `WHERE primary_key IN (chunk_i)` — same schema (field space), restricted row set (index subspace). Each chunk is independently addressable, partition-able again, and usable in any context where a logical region is expected. Where MPI codes carve up data with offsets and counts, Legion gives you a typed handle for each chunk.

## Mechanism & API
**Reaching a subregion**:
```cpp
LogicalPartition lp = runtime->get_logical_partition(ctx, lr, ip);
LogicalRegion subregion = runtime->get_logical_subregion_by_color(ctx, lp, DomainPoint(color));
```

`LogicalPartition` is the runtime handle representing all subregions of `lr` partitioned by `ip`; `get_logical_subregion_by_color` resolves one specific color.

**Inside an `index-space-launch.md`**, subregions are reached via a `projection-functor.md`:
```cpp
IndexLauncher launcher(TASK_ID, color_space, ...);
launcher.add_region_requirement(
    RegionRequirement(lp, /*projection_id=*/0, READ_WRITE, EXCLUSIVE, lr));
// Each point task `i` receives subregion lp[i].
```

**Partition a subregion further** (hierarchical partitioning):
```cpp
IndexPartition sub_ip = runtime->create_equal_partition(ctx,
    runtime->get_index_subspace(ctx, ip, DomainPoint(color)),
    sub_color_space);
```

**Hierarchical partitioning paper** (`oopsla2013.pdf`) describes this pattern in depth.

## Invariants
- A subregion's `index-space.md` is a subset of its parent's; its `field-space.md` is identical to its parent's.
- A subregion is itself a `logical-region.md` with the same `tree_id` as its parent (siblings share the tree ID; distinct top-level regions have distinct tree IDs).
- Subregions are **materialized lazily** — the runtime allocates the data structures for them only when used.
- A subtask requesting a subregion via a `RegionRequirement` is bound by the **subset privilege rule**: the parent task must hold privileges on a region that contains this subregion.
- Subregion of subregion of … forms an arbitrarily-deep tree (paper `oopsla2013.pdf`).

## Performance implications
- **Materialization is lazy and cheap**; partitions of millions of subregions cost only the partition structure, not per-subregion overhead.
- Each subregion's **physical instance** is allocated independently by `map-task.md`; the mapper can place different subregions in different memories.
- The `equivalence-set.md` machinery tracks per-subregion state; deeply-fragmented subregion hierarchies can inflate equivalence-set counts.
- For data-parallel work, subregions enable per-point physical instances that fit on per-processor caches — usually a big win vs. one giant parent instance.

## Debug signals
- **Error 105** ("Region not a subregion"): an operation required a subregion relationship that doesn't hold. Check the partition handle and color.
- **`legion-spy.md`** region-tree output renders the subregion hierarchy; use it to confirm the structure matches the application's intent.
- **`-DPRIVILEGE_CHECKS`** catches subset-privilege violations between a subtask's subregion request and the parent's privilege.

## Failure modes
- Using a subregion handle after its `LogicalPartition` is destroyed → undefined behavior.
- Requesting a subregion that doesn't exist (wrong color) → runtime error.
